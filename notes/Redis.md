
### 🚨 灾难现场重现：锁被别人偷了！
假设你的锁 30 秒过期，但你（线程 A）慢吞吞地执行了 35 秒。
1. **第 30 秒：** Redis 说“时间到！”，无情地自动把你的锁释放了。
2. **第 31 秒：** 第二个人（线程 B）一看门没锁，大摇大摆地拿到了锁进来了。
3. **数据错乱：** 此时，你和线程 B 都在修改同一份订单数据，绝对会把数据改坏！
4. **更可怕的连环车祸：** 第 35 秒你终于执行完了，你顺手跑了一句 `del lock`（释放锁）……结果你把**线程 B 刚加上的锁给删了**！接着线程 C 发现没锁又进来了……无限恶性循环！

### 🛡️ 满分救场方案（两套组合拳）



**绝招 1：看门狗（Watch Dog）自动续期**
也就是你简历里用到的 **Redisson** 框架自带的黑科技。如果你不传过期时间，Redisson 会默认给你加一个 30 秒的锁，并在后台悄悄派一只“看门狗”（定时任务）。
只要你的业务还没执行完，看门狗就会每隔 10 秒去 Redis 里把锁的时间重新拉回到 30 秒。**只要你不死（服务器没宕机），锁就绝对不会提前释放！**

**绝招 2：解铃还须系铃人（唯一标识 + Lua 脚本）**
为了防止第 35 秒你执行完去误删别人的锁，加锁的时候不能随便写，要把 Value 设置成**你自己的专属 UUID（或线程 ID）**。
在删锁的时候，先确认一下：“这把锁的 UUID 是我的吗？”如果是，再删。
注意！“判断”和“删除”是两个动作，为了防止在这两个动作之间别人插队，必须把它们写成一段 **Lua 脚本**，交给 Redis 一气呵成地执行（保证原子性）。

# 🚀 Redis 核心高频考点深度总结

## 1. 缓存三大坑及解决方案
| 问题 | 现象描述 | 核心原因 | 常用解决方案 |
| :--- | :--- | :--- | :--- |
| **缓存穿透** | 查询一个**根本不存在**的数据，Redis 和 MySQL 都查不到。 | 恶意攻击或系统 Bug | 1. **布隆过滤器 (Bloom Filter)** 拦截。<br>2. 缓存空对象 (设极短过期时间)。 |
| **缓存击穿** | 某个**极度热点**的 Key 突然过期，导致瞬间海量请求打爆数据库。 | 热点 Key 刚好失效 | 1. **互斥锁 (分布式锁)**，只放行一个请求去查 DB。<br>2. **逻辑过期**（后台异步更新，前端先用旧数据）。 |
| **缓存雪崩** | **大量**的 Key 在同一时间点同时过期，或 Redis 直接宕机。 | 集中过期 / 服务宕机 | 1. 过期时间加上**随机扰动值**（如 1~5 分钟）。<br>2. 搭建高可用集群 (哨兵/Cluster)。<br>3. 服务降级限流。 |

## 2. Redis 分布式锁的核心要素
实现一个健壮的分布式锁，必须满足以下几个条件：
1.  **互斥性**：使用 `setnx` (SET if Not eXists) 保证只有一个线程能抢到锁。
2.  **防死锁**：必须给锁设置合理的**过期时间**（`expire`），防止持有锁的节点宕机导致死锁。而且 `setnx` 和设置过期时间必须是**原子操作**（一条命令完成）。
3.  **防误删**：锁的 Value 必须设置为当前线程的**唯一标识**，释放锁前判断是否为自己的锁。
4.  **原子释放**：判断和删除锁的动作必须使用 **Lua 脚本**执行，保证原子性。
5.  **自动续期**：使用 **Redisson 的 Watch Dog 机制**，后台定时为未执行完的业务自动续期，防止锁提前释放导致并发冲突。
